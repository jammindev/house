## Router basename → URL name mapping

| Model | Router basename | URL names |
|---|---|---|
| ElectricityMeter | `electricity-meter` | `electricity-meter-list`, `electricity-meter-detail` |
| MeterReading | `electricity-meter-reading` | `electricity-meter-reading-list`, `electricity-meter-reading-detail` |
| ConsumptionSummaryView | — (standalone APIView) | `electricity-consumption-summary` |

## Duplicate reading_at rejection

The serializer has a custom check in `MeterReadingSerializer.validate()` that raises `{"reading_at": "..."}`. However, the DB `UniqueConstraint` on `(meter, register, reading_at)` can surface first as `{"non_field_errors": [...]}` depending on race conditions. Tests assert `"reading_at" in data or "non_field_errors" in data` to handle both paths.

**Why:** The serializer path only runs within the serializer's validate method; the DB path fires if two concurrent requests bypass the serializer check.

## ConsumptionRecord source=reading never comes from factories

Records with `source=ConsumptionSource.READING` are exclusively generated by `rebuild_reading_records()` which runs inside `perform_create/update/destroy` of `MeterReadingViewSet`. Factories write raw rows. Tests that assert on reading-derived records must create readings through the API (POST). See [[feedback_factory_vs_api_for_side_effects]].

## Permission model

`IsElectricityOwnerWriteMemberRead`: SAFE_METHODS (GET, HEAD, OPTIONS) → any household member; writes (POST/PATCH/PUT/DELETE) → household owner only. ConsumptionSummaryView shares this permission class.

## Household activation pattern

Tests activate the household on the user: `user.active_household = household; user.save(update_fields=["active_household"])`. The `ActiveHouseholdMiddleware` reads `user.active_household` to set `request.household`. Cross-household isolation relies on the queryset filter `filter(household=selected_household)`.

## ConsumptionImport viewset conventions

- `http_method_names = ["get", "post", "head", "options"]` — DELETE and PUT return 405.
- POST always returns 201. A business failure (unrecognized format, parse error) produces `status='failed'` on the trace object, not an API-level error. Zero `ConsumptionRecord`s are written on failure.
- Idempotence: re-importing the same file via `ignore_conflicts=True` on `bulk_create` → `created_count=0`, `skipped_count=N`.
- `options` field is a JSON string in multipart form data — the view decodes it with `json.loads`; malformed JSON → 400.
- `provider` must be a key present in `importers.REGISTRY`; unknown value → 400 on `"provider"`.
- `preview` is a custom action at `POST .../preview/`. Since it's a POST it goes through `IsElectricityOwnerWriteMemberRead` → members get 403, owner gets 200.
- Test fixture: `apps/electricity/tests/fixtures/enedis_courbe_de_charge.csv` produces exactly 5 ConsumptionRecords (6 rows, 1 empty Valeur skipped).

## Honesty rule in consumption_summary

`granularity="hour"` excludes records with `interval_minutes > 60`; `granularity="day"` excludes records with `interval_minutes > 1440`. `month` and `year` have no cap (`max_interval=None`). A 1440-min reading-derived daily record is invisible in the hour view but visible in day/month/year.
